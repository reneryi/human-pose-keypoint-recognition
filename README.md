# 基于 Git 与 GitHub 的人体关键点识别软件版本管理实验

## 任务一：认识版本管理与 Git 基础

### 1. 为什么软件开发需要版本管理

在软件开发过程中，代码会随着需求变化、功能增加、问题修复和测试反馈不断修改。如果只通过复制文件夹的方式进行备份，容易出现以下问题：

1. 文件版本混乱，难以判断哪个文件夹是最新版本。
2. 无法清楚知道每次修改了哪些内容。
3. 出现错误后，难以精确回退到某一个稳定版本。
4. 多个功能同时开发时，容易互相影响。
5. 不方便记录开发过程，也不利于后续项目提交和检查。

Git 作为分布式版本控制系统，可以完整记录项目从创建、开发、测试到发布的过程。每一次提交都可以保存当前代码状态，并通过提交信息说明修改原因。这样不仅方便代码管理，也便于展示软件开发过程。

---

### 2. Git 能解决的问题

Git 在本项目中主要用于解决以下问题：

1. **版本回退**  
   当代码修改后出现错误时，可以根据提交记录回退到之前的稳定版本。

2. **历史追踪**  
   Git 可以记录每次提交的作者、时间、修改内容和提交说明，方便查看项目演化过程。

3. **功能隔离**  
   可以通过分支开发不同功能，例如在 `feature` 分支中开发视频人体关键点识别功能，避免影响主分支中的稳定代码。

4. **多人协作**  
   虽然本项目以个人完成为主，但 Git 支持多人同时开发、提交、合并代码，是实际软件工程协作的重要工具。

5. **版本发布**  
   可以通过标签 `tag` 标记稳定版本，例如 `v1.0-image` 表示图片识别版本，`v2.0-video` 表示视频识别版本。

---

### 3. Git 的基本区域和对象

Git 的基本工作流程涉及以下几个区域：

1. **工作区**  
   当前正在编辑和修改的项目文件所在区域。

2. **暂存区**  
   使用 `git add` 后，文件会进入暂存区，表示这些修改准备被提交。

3. **本地仓库**  
   使用 `git commit` 后，暂存区中的内容会保存到本地 Git 仓库中，形成一次提交记录。

4. **远程仓库**  
   远程仓库用于保存和共享项目代码。本课程要求中以 Gitee 为例，本项目实际使用 GitHub 作为远程代码托管平台，二者在 Git 操作流程上基本一致。

---

### 4. 常用 Git 命令说明

| 命令 | 作用 |
| --- | --- |
| `git init` | 在当前目录初始化一个新的 Git 仓库 |
| `git clone` | 从远程仓库克隆项目到本地 |
| `git status` | 查看当前工作区和暂存区状态 |
| `git add` | 将修改添加到暂存区 |
| `git commit` | 将暂存区内容提交到本地仓库 |
| `git log` | 查看提交历史记录 |
| `git diff` | 查看文件修改前后的差异 |
| `git branch` | 查看、创建或删除分支 |
| `git checkout` | 切换分支或恢复文件 |
| `git switch` | 切换分支，是较新的分支切换命令 |
| `git merge` | 合并指定分支到当前分支 |
| `git pull` | 从远程仓库拉取最新代码并合并 |
| `git push` | 将本地提交推送到远程仓库 |
| `git tag` | 创建、查看或管理版本标签 |

## 任务二：本地仓库建立与文件基本操作

### 1. 本地仓库建立方式

本项目首先采用本地初始化的方式建立 Git 仓库。在项目目录中执行以下命令：

```bash
git init -b main
```

该命令会在当前项目目录下创建 `.git` 隐藏目录，并将当前目录纳入 Git 版本管理。本项目主分支命名为 `main`。

### 2. 文件基本操作

任务二要求在本地仓库中完成文件的新建、修改、差异查看、删除或重命名，并将这些变化纳入版本管理。本项目中的对应操作计划如下：

1. 新建并提交 `README.md` 项目说明文档。
2. 修改已有的 `README.md`，补充任务二说明内容。
3. 使用 `git status` 查看工作区与暂存区状态。
4. 使用 `git diff` 查看 README 修改前后的差异。
5. 新建 Git 操作记录文档，并通过重命名操作体现文件变更管理。
6. 使用 `git log` 查看提交历史。

### 3. 远程仓库与克隆方式

课程要求中以 Gitee 为例创建远程仓库。本项目实际使用 GitHub 创建远程仓库，仓库名称为：

```text
human-pose-keypoint-recognition
```

在完成本地仓库提交后，将本地 `main` 分支推送到 GitHub 远程仓库。随后再使用 `git clone` 将远程仓库克隆到本地临时目录，以体现第二种仓库建立方式。

### 4. 提交记录要求

本项目不会一次性提交全部内容，而是按照功能和文档变化形成多次提交。任务二阶段至少形成以下提交记录：

1. 初始化项目说明文档。
2. 补充本地仓库建立说明。
3. 添加许可证文件。
4. 添加并重命名 Git 操作记录文档。

这些提交记录可以通过以下命令查看：

```bash
git log --oneline
```

## 任务三：软件版本 v1.0.0 开发

### 1. 版本目标

`v1.0.0` 是本项目的第一个可运行软件版本，主要实现图片人体关键点识别功能。该版本使用 MediaPipe Pose 预训练模型，不需要自行训练模型即可完成单张图片中的人体关键点检测。

本版本实现的功能包括：

1. 支持通过命令行指定输入图片。
2. 支持通过本地文件选择窗口选择图片。
3. 使用 MediaPipe Pose 识别人体主要关键点。
4. 使用 OpenCV 在图片上绘制人体关键点和骨架连线。
5. 将识别结果保存到输出目录。
6. 对未检测到人体的图片给出友好提示，并保存原图作为结果图。

### 2. 项目结构

任务三阶段新增的主要文件结构如下：

```text
├── main.py                         # 命令行版图片识别入口
├── app_image.py                    # 本地图片选择版入口
├── requirements.txt                # Python 依赖说明
├── src/
│   ├── __init__.py
│   └── image_pose_estimator.py     # 图片人体关键点识别核心逻辑
├── assets/
│   ├── input/                      # 示例输入图片目录
│   └── output/                     # 预留输出资源目录
└── outputs/                        # 程序默认结果输出目录
```

### 4. 命令行运行方式

将待识别图片放入 `assets/input/` 目录，例如：

```text
assets/input/person.jpg
```

然后执行：

```bash
python main.py --input assets/input/person.jpg --output outputs/person_result.jpg
```

如果不指定 `--output` 参数，程序会自动将结果保存到 `outputs/` 目录，例如：

```bash
python main.py --input assets/input/person.jpg
```

默认输出文件名类似：

```text
outputs/person_pose_result.jpg
```

### 5. 本地文件选择运行方式

如果希望通过图形窗口选择图片，可以执行：

```bash
python app_image.py
```

程序会弹出文件选择窗口，选择图片后自动完成识别，并在完成后弹窗提示结果图片保存位置。

当图片人体关键点识别功能测试通过后，在主分支 `main` 上发布版本标签：

git push origin v1.0.0

`v1.0.0` 表示本项目第一个稳定版本，即图片人体关键点识别版本。
